作成日: 2026-06-19
目的: 自分以外（テラ従業員・スキルスリー）が使える共有版にする。
ねらい: **誰が・何を・どこまでやっているかを可視化し、自動化できるところは全てAIで。**
※本書は設計と見積もりのみ。実装・デプロイは別途GOで着手（掟 第3条）。

## 1. 採用スタック（確定＝完全無料構成 / AIは各自のClaude）

**方針決定**: サーバー側でClaude APIを叩く有料自動化はやらない。AIは**各メンバー自身のClaude**が担う（＝我々のAI課金 $0）。インフラも無料枠で組む。

**手持ちアカウント＝Supabase + GitHub だけで完結**（Cloudflare不要）。さらにP1のCRUDは Supabase の自動API（PostgREST）を `supabase-js` で直接叩くので、**専用APIサーバすら不要**。

| レイヤー | 採用（Supabase + GitHub のみ） | 補足 |
|---|---|---|
| 認証・組織 | **Supabase Auth** ＋ 組織は `workspace` テーブルで自作 | テラ/スキルスリーを分離 |
| DB＋API＋リアルタイム | **Supabase（Postgres＋自動REST＋RLS＋Realtime）** | P1は`supabase-js`で直接続。APIサーバ不要 |
| フロント | **GitHub Pages**（無料・商用OK） | 既存Vite/React SPAを配信 |
| MCP（P2） | **Supabase Edge Functions（Deno）** で Streamable HTTP + OAuth | 各ユーザーが自分のClaudeに接続 |
| リポジトリ/CI/デプロイ | **GitHub（+ Actions）** | Pagesへ自動デプロイ |
| AI / 自動化 | **各メンバー自身のClaude** | サーバー課金ゼロ |

> 両サービスとも無料枠。独自ドメインは任意（年$10程度、`*.github.io`なら$0）。

## 2. マルチテナントの基本設計

- 全データに `workspace_id` を付与し、**行レベルで完全分離**（テラのデータはスキルスリーから不可視）。
- 1ユーザーは複数ワークスペースに所属可（Clerk Organizations と対応）。
- ロール: `owner` / `admin` / `member`。可視・編集・割当の権限を分ける。

```
User ──< Membership(role) >── Workspace（テラ / スキルスリー …）
                                  │
                    ┌─────────────┼─────────────┐
                  Goal           Task          Activity
              (担当・親子・進捗)  (assignee・状態) (誰が何をしたか)
```

---

## 3. データモデル（可視化の肝）

| テーブル | 主な項目 | 可視化での役割 |
|---|---|---|
| workspace | id, name | テラ/スキルスリーの箱 |
| membership | user_id, workspace_id, role | 誰がどのチームか |
| goal | workspace_id, owner_id, parent_id, title, horizon, criteria, **progress** | ゴール階層と進捗 |
| task | workspace_id, goal_id, **assignee_id**, title, **status**(todo/doing/done/blocked), due | 誰が何を・状態 |
| activity | workspace_id, actor_id, ts, type, summary | 誰が・いつ・何をした |
| comment | task_id/goal_id, author_id, body | 経緯・詰まり |

→ **チームダッシュボード**で「鈴木：提案資料80%／山田：競合調査で blocked」が一目で出る。

---

## 4. MCPのマルチテナント化（動画の「蒼太君に振る」を実現）

- OAuthトークンが `(user, workspace, role)` を識別。全ツールが自動でそのチームに絞り込み。
- 追加ツール：
  - `list_members` … メンバー一覧
  - `get_member_progress(member)` … 「誰が何をどこまで」
  - `assign_task(to, ...)` … 振ると相手のClaudeが読む
  - `list_team_activity` … チームの活動フィード
- 既存9ツールはテナント対応に改修（workspace_idで絞る）。

---

## 5. 自動化（AIは各自のClaudeが担う＝サーバー課金ゼロ）

サーバーがClaude APIを叩くのではなく、**各メンバーが自分のClaudeに話しかけて**MCP経由で実行する：
- 各自が朝「進捗まとめて」→ 自分のClaudeがチーム状況を集約
- 「滞ってるタスク教えて」→ blocked/期限超過を抽出
- 「蒼太に振っといて」→ `assign_task` で相手に割当（相手のClaudeが受け取る）
- レポートも各自のClaudeに生成させる

→ アプリ（サーバー）側は**データの保管・共有・MCP公開**に徹し、AIコストを持たない。
（将来どうしてもサーバー常駐の自動化が要れば、そこだけ従量課金で追加可能。今はやらない。）

## 8. 主なリスク・論点

- **MCPのOAuth実装**が最難関（本物も /authorize /token /register を自前。Clerkで人間ログインを裏に）。
- 権限設計（どのロールが誰の進捗を見えるか）は要合意。
- 自動割当のAI品質（誤割当を防ぐ承認フロー）。
- デプロイ＝公開は掟 第3条によりGO必須。
