# 🛡️ プロジェクトの掟（CONSTITUTION）

このファイルは本プロジェクト（**TERA** ＝ 目標達成型タスク管理 ＋ AIアシスタント）の最上位ルール。
実装・リファクタ・提案のすべてはこの掟に従う。迷ったら掟を優先する。
掟に反する変更は、たとえ動いても「不正解」とみなす。

---

## 第1条: App.jsx は「司令塔」に徹する

- `src/App.jsx` はアプリ全体の **オーケストレーター（司令塔）** であり、ロジックの実装場所ではない。
- **目安は 388 行前後**。これを大きく超えたら「責務が漏れ出している」サイン。
  hooks / components / routing へ切り出してスリム化する。
- App.jsx がやってよいこと:
  - 全体の状態・依存の組み立て（hooks の呼び出し）
  - 画面の組み立て（components / routing の配置）
  - トップレベルの文脈（プロバイダ、レイアウト枠）の提供
- App.jsx がやってはいけないこと:
  - 個別の業務ロジック（API呼び出し、計算、整形）の直接記述 → `hooks/` `utils/` へ
  - 込み入ったJSXの直接記述 → `components/` へ

## 第2条: ロジックとUIを完全に分離する

- **ロジック = `src/hooks/`**: 状態管理・副作用・API・ビジネスルールはカスタムフックに集約。
  - 例: `useGoals()`, `useTodayTodo()`, `useChat()`, `useAddy()`
- **UI = `src/components/`**: 見た目と表示だけ。データは props か hooks 経由で受け取る。
  - components 内に fetch / 業務計算を書かない。「受け取って描くだけ」。
- **純粋関数 = `src/utils/`**: 日付整形・バリデーション・変換など、状態を持たない処理。
- **画面遷移 = `src/routing/`**: ルート定義・画面の出し分けのみ。
- 原則: 「**この処理はどのフォルダの責務か**」を毎回自問してから書く。

## 第3条: データは Supabase ＋ RLS で守る。秘密情報を漏らさない

- データ保護の主役は **Supabase の RLS（行レベルセキュリティ）**。クライアント側のチェックは
  あくまで体験向上であり、防御ではない。**新しい表を足したら必ず RLS を設計する**。
- フロントに置いてよいのは **publishable / anon キーまで**（公開前提）。
  `service_role` キー・`SUPABASE_ACCESS_TOKEN`・JWT秘密鍵などを **フロントや Git に出さない**。
- 秘密情報は `.env`（gitignore 済み）に置く。`.env` をコミットしない。
- MCP（ホスト版）はプロジェクト新方式キー（`sb_publishable_…`）＋ **ES256 非対称署名**。
  ユーザーとして動くトークンは自前署名できない（秘密鍵を持てない）ため、
  **Supabase が署名した生トークンを使う／キャッシュする**方式に従う。

## 第4条: MCP は「話すだけで仕事が終わる」の心臓部

- TERA の本質は、Claude のコネクター（MCP）経由でゴール・タスク・活動記録を読み書きできること。
- MCP は2系統あり、役割を混同しない:
  - **ローカル STDIO**（`mcp/server.js`）: 開発・自分用。HTTPS/OAuth 不要。
  - **ホスト版**（`supabase/functions/mcp/`）: 本番。OAuth2.1+PKCE+DCR で「URL貼付コネクター」を成立させる。
- ツールの権限は**サーバ側 RLS で保証**する。クライアント（Claude）を信用しない。
- 「おはよう／仕事しよう」等の合図では、まず `get_context` / `list_tasks({todayOnly}) ` / `list_goals` で文脈を取る。

## 第5条: デスクトップとスマホ、両方で成立させる

- このアプリは **PCでもスマホでも使う**。どちらかを犠牲にしない。
- レスポンシブは **モバイルを既定** にし、`sm:` 以上で従来のデスクトップ表示を保つ
  （例: `px-4 py-6 sm:px-10 sm:py-8`）。**既存のデスクトップ表示を壊す変更は不正解**。
- 固定幅（`w-[340px]` 等）や多カラムは、狭い画面での挙動を必ず考える。

## 第6条: リリース＝デプロイは人間のGOを得てから

- **フロント（GitHub Pages）は `main` への push で自動デプロイ**される
  （`.github/workflows/deploy.yml` → Actions → Pages、`base=/tera-app/`）。
  したがって `main` への **push はそのまま本番リリース**。push は人間の明示指示があるまで行わない。
- **Supabase（Edge Function `mcp` / DBマイグレーション）の反映は人間が実行**する
  （`npx supabase functions deploy mcp` / SQL Editor or Management API）。Claude は勝手にデプロイしない。
- ローカルの `dev` 起動・`npm run build` での確認は可。破壊的操作（force push、データ削除、DDL）の前は必ず確認する。

## 第7条: 整合性は「導出」と「DB制約」で守る（コピーを撒かない）

- **表示する値は、保存したコピーではなく“生きてる実体から都度導出”する。** 名前・タイトル・件数・進捗・「何が起きたか」を別テーブルに焼き付けない（リネーム/削除で陳腐化＝ゴーストの元）。
  - 例: 「直近の動き」は活動ログのフリーテキストではなく tasks/goals から導出。進捗は `goals.progress` ではなくタスク完了から算出。
- **不整合は DB が物理的に拒否する。** 必須・非空は `CHECK`/`NOT NULL`、参照は `FK`（cascade か set null かを意図して選ぶ）。入口（MCP/フロント）でも検証し二重防御にする。
- **新しく「保存して別所で表示する値」を足さない。** 監査ログ等がどうしても要るなら、フリーテキストではなく**実体参照（target_id）＋読み取り時に現在値で描画**にする。
- 迷ったら「この値は実体から導出できるか？」を先に問う。できるなら保存しない。

## 補足ルール

- 1ファイル1責務。肥大化したら割る。
- 命名は役割が分かる名前にする（`utils.js` のような何でも箱を作らない）。
- 既存コードのスタイル（命名・コメント量・粒度）に合わせる。
- コメントは「何を」より「なぜ」を残す（既存コードのトーンに合わせる）。
- 不明点は掟に照らして判断し、それでも決まらなければ人間に聞く。
