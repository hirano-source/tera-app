# Addness クローン 要件定義書

作成日: 2026-06-19
目的: スクリーンショットから読み取った「Addness」アプリと同等／ほぼ同等のアプリを構築する

---

## 1. アプリ概要

AIアシスタント **"Claude"** を中核に据えた、**目標達成型タスク管理 ＋ チームチャット** のWeb SaaS。

- 個人/チームが「ゴール」を設定し、AIが分解・問い・今日やるべきToDoを提案する
- ゴールごとにチャットスレッド・リソース(ファイル)・ハドル(音声)が紐づく
- カレンダー連携で今日の予定をToDoに取り込む

想定ワークスペース例: 「ジブンゴルフ」（=組織/チーム単位で区切る）

---

## 2. 画面・機能一覧

### 2.1 共通レイアウト
- 左サイドバー: 実行(ToDo) / ゴール / チャット / 通知 / メンバー / 設定
  - 各アイコンに未読・件数バッジ（例: 実行に「1」）
- 上部グローバル: 検索バー、ポイント(コイン)表示
- 右下: Claudeからのトースト通知（「Claudeが今日のゴールから問いを用意しました」）

### 2.2 今日のToDo `/todo`
- 「今日のToDo」リスト（Beta）。手動追加(＋)、選び直し
- 「Claudeと今日のToDoを決める」: AIがゴールから今日やるべきToDoを選定
- 「今やるべきゴール」セクション: ゴールを並べ、Claudeに相談ボタン
- カレンダー連携した予定を当日のToDoに表示

### 2.3 カレンダー連携（モーダル）
- iCal/ICS URLを複数登録（Googleカレンダー等）
- 当日の予定を今日のToDoに反映
- 完了状態のみ保存、カレンダー本体には書き戻さない（読み取り専用同期）

### 2.4 ゴール `/goals/{id}`
- ゴール（階層構造: 親ゴール → 配下リソース/サブゴール）
- メタ情報: 現状 / 完了の基準 / 期日 / 担当メンバー
- サブタスク: 「〜する」形式で入力
- リソース: ファイルのアップロード(D&D)/新規作成、検索・並び替え、配下リソースのツリー
- ハドルを開始: 音声ミーティング
- コメント機能（件数表示）
- 「ゴールについて相談」: Claudeに相談

### 2.5 チャット `/chat`
- ワークスペース名表示
- タブ: 今日のToDo / ゴール / DM / グループ
- ゴールを選択するとスレッドが表示される（ゴール紐付きスレッド）
- DM・グループチャット

### 2.6 通知 `/notifications`
- フィルタ: 未読 / すべて、種類フィルター、並び替え(新しい順)
- 一覧 + 詳細の2ペイン

### 2.7 メンバー `/members`
- メンバー招待・一覧・権限

### 2.8 設定 `/settings`
- ワークスペース/プロフィール/カレンダー連携 等

### 2.9 Claude（AIアシスタント）— 横断機能
- ゴールの分解・相談
- 今日のToDo選定
- ゴールから「問い」を生成し通知
- 各画面の「相談」起点

### 2.10 ゴール／タスクの設計思想（運営者メソッド）

このアプリの芯。UI・データモデル・AIプロンプトはこの思想に従う。

**ゴールの階層（時間ネスト）**
```
7年後ゴール → 1年後ゴール → 今月の目標 → 今日のタスク（今日のToDo）
```
上位ゴールを噛み砕いて下ろす。「今日のToDo」は上位ゴールからの派生物。
→ ゴールは親子の階層構造を持つデータモデルにする。

**ゴール設定フロー（「Claudeに相談」で支援する中身）**
1. 葬式で言われたい言葉をイメージして出す
2. それを「〜なことは価値のある賛すべきことだ」という**価値観に変換**（このステップをClaudeが代行）
3. 価値観を**7年後の数値目標**に伸ばす。数値は「行けそう」と直感で思えるラインで決める
4. 7年→1年→今月→今日へブレイクダウン（各段で上位達成可能性の検証も兼ねる）

**順算と逆算（「今日のToDoを決める」ロジックの指針）**
- 逆算可能 → 上位目標から砕いて今日のタスクを出す
- 逆算不能 → 順算＝「今取りうる中で一番良さそうな1手」を1つだけ出し、結果から学ぶ

**タスクの本質**: 頭の中のデータを外に出し、1タスクに集中させる装置。
AIに投げる時も「タスク名を1つきっちり作り、1つだけ実行させる」と精度が上がる。

## 3. データモデル（概略）

- User（ユーザー）
- Workspace（組織）/ Membership（所属・ロール）
- Goal（ゴール: 親子・現状・完了基準・期日・担当）
- Task / Todo（サブタスク、今日のToDo、出所=goal or calendar or manual）
- Resource（ファイル/成果物、ゴール配下のツリー）
- Thread / Message（チャット: goal紐付け / DM / group）
- Notification（通知）
- CalendarSource（iCal URL）/ CalendarEvent
- Point（ポイント）
- ClaudeConversation / Prompt（AI対話ログ）

---

## 4. 非機能要件
- 認証（メール/OAuth）、ワークスペースのマルチテナント
- リアルタイム（チャット・通知・ハドル）
- ファイルストレージ
- レスポンシブ（スクショはデスクトップ幅）
- 日本語UI

---

## 5. 実現可能性の判定（サマリ）

| 機能群 | 難易度 | 備考 |
|---|---|---|
| レイアウト/ナビ/各画面UI | 低 | 標準的なCRUD UI。確実に作れる |
| 認証/ワークスペース/メンバー | 低〜中 | 既存ライブラリで対応可 |
| ゴール/タスク/リソース管理 | 中 | 階層・ファイルストレージあり |
| カレンダー連携(iCal取込) | 中 | ICSパース＋定期取得 |
| チャット(スレッド/DM/グループ) | 中〜高 | リアルタイム基盤が必要 |
| 通知 | 中 | リアルタイム配信 |
| Claude(AI) | 中 | Claude APIで実装可。プロンプト設計が肝 |
| ハドル(音声) | 高 | WebRTC/SFU。MVPでは後回し推奨 |

**結論: 技術的に実現可能。** ただし全機能フルスコープは規模が大きいため、段階的構築を推奨。
